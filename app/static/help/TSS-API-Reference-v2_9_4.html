API Reference v2.9.4

TSS Declaration API

The Trader Support Service (TSS) Declaration API allows programmatic creation, updating, submission and retrieval of ENS, SFD, Supplementary, Full Frontier and Internal Market Movement declarations for goods moving to Northern Ireland.

📅 Published: February 2026
🔗 Protocol: HTTPS REST / JSON
🔐 Auth: Basic Authentication
📋 Format: application/json
Authentication & Protocol

All API calls use HTTPS with Basic Authentication. Credentials follow the pattern API.TSS0010026. The username and password must be Base64-encoded and passed in the Authorization header.

Required HTTP Headers

Accept
application/json
Content-Type
application/json
Authorization
Basic <Base64(username:password)>
Time Zones: All date/time fields (e.g. arrival_date_time) use UTC. Times displayed in the portal may differ by 1 hour during BST.
Attachments: Attachments are not permitted and will be ignored or may cause the transaction to be rejected.

Schema Overview

Below is a conceptual overview of how the major data models relate to each other in the TSS system.

ENS Declaration Header

Top-level transport record. Ref prefix: ENS…. Contains transport, carrier, and routing details. One header can contain many consignments.

Consignment (ENS)

Child of a Declaration Header. Ref prefix: DEC…. Contains consignor, consignee, importer, exporter and SFD-creation details.

Goods Item

Child of a Consignment. Ref: 32-char hex. Up to 99 per consignment. Applicable to ENS, SFD, SD, FFD and IMMI.

SFD Header

Standalone SFD header. Ref prefix: SFD…. Used for standalone SFD journeys not initiated via ENS.

Simplified Frontier Declaration

Created from an ENS consignment or standalone. Ref prefix: DEC…. Links to a Supplementary Declaration.

Supplementary Declaration

Ref prefix: SUP…. Links to an SFD. Contains duty/payment information. Can be standalone or ENS-initiated.

Full Frontier Declaration (FFD)

Ref prefix: FFD…. Supports H1–H4 declaration categories. Covers maritime and non-maritime movements.

Maritime Inventory Claim

Ref prefix: ICR…. Links containers and bills of lading to consignments.

GVMS GMR

Goods Movement Record. Ref prefix: GMR…. Links consignments under a single border-crossing vehicle reference.

Internal Market Movement (IMMI)

Ref prefix: GLR…. Used for goods moving GB→NI under UKIMS simplified process. Supports H1/H8 reclassification.

Standard Response: All successful POST operations return {"result": {"status": "created|updated|cancelled|submitted|deleted|reclassified", "process_message": "SUCCESS", "reference": "<ref>"}}. Errors return status: "error" with a process_message describing the issue.

Error Handling

Successful requests return HTTP 20x. Errors return appropriate HTTP error codes. All error payloads include status: "error" and a descriptive process_message.

{ "result": { "status": "error", "process_message": "ERROR: Invalid op_type for 'supplementary_declarations': updated", "error_details": "" } }
{ "result": { "status": "error", "process_message": "ERROR: Mandatory field 'controlled_goods' not supplied", "error_details": "" } }
{ "result": { "status": "error", "process_message": "ERROR: Unable to access target record: DEC100000000022711", "error_details": "" } }
{ "result": { "status": "error", "process_message": "ERROR: Invalid format for 'goods_domestic_status' rejected: XXXXX", "error_details": "" } }
Important: All payload fields are validated before any processing. If anything is incorrect in the payload, an error is returned and nothing is processed. Fields not mentioned in the specification are silently ignored.
