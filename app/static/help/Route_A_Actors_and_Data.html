What's master data, what's transactional?

The more we can lock down as master data, the less a customer has to supply per shipment — and the more of the declaration we can auto-populate. The split below is how Synovia Flow models it for Clarity.

Master Data

Configured once, reused across every movement.
CADENCE • SET UP AT ONBOARDING, REVIEWED QUARTERLY

Parties & identifiers

  • Customer master — name, EORI (GB + XI), TSS number, addresses
  • Carrier master — carrier name (35-char max), carrier EORI (XI)
  • Consignor / Consignee registry — trading partners and their EORIs
  • Clarity agent record — agent identity for declarations

Product data

  • Product Master — SKU, description, 10-digit commodity code, net weight per unit, supplementary units, country of origin, preference eligibility
  • Recurring invoice templates — fingerprinted layouts per customer, used by the ingestion engine

Route & transport defaults

  • Route profiles — carrier × port × movement type combinations
  • Arrival ports — e.g. GBAUBELBELBEL (Belfast Port)
  • Movement type defaults — 3a for RoRo, container indicator rules
  • Transport charges rules — "account holder with carrier" flags per customer

Reference data

  • Commodity code library — HMRC Trade Tariff cache, refreshed periodically
  • CPC codes — customs procedure classifications
  • Currency codes — ISO 4217
  • Country codes — ISO 3166-1 alpha-2
  • Incoterms codes — 2020 edition: EXW, FOB, CIF, DDP, DAP, etc.

Access & compliance

  • User accounts — declarants, approvers, customer-portal users
  • TSS API credentials — per environment (PRD, TST)
Per movement

Transactional Data

Unique to each shipment — ingested from customer inputs.
CADENCE • CAPTURED LIVE, ONE DATASET PER DECLARATION

Booking & movement

  • Movement reference — unique ID for the shipment
  • Booking reference — carrier's booking no.
  • ETA / ETD — in DD/MM/YYYY HH:MM:SS format (UTC)
  • Trailer / container ID — per physical unit
  • GMR reference — issued after GMR submit

Consignment & goods

  • Sales order — customer PO / job number, delivery ref
  • Dispatch / pack list — actual quantities moved this trip
  • Line-level weights — net + gross for this shipment
  • Package count — number of packages, packaging type

Commercial

  • Commercial invoice — invoice number, date, value per line
  • Currency + FX context — per invoice
  • Preference claim — if applicable, per goods line
  • Incoterms + place — delivery terms for this shipment

TSS API outputs

  • ENS goods IDs — stage-specific hex, returned by TSS
  • SFD reference — auto-generated by TSS
  • SDI goods IDs — different hex values to ENS
  • API response log — per call, written to client schema

Exceptions

  • Missing-data flags — fields the engine couldn't auto-resolve
  • Portal update tokens — one-time links sent to customers

Why this split matters for automation

Less to ingest

Every field that lives in master data is a field we don't need to parse from an email or PDF. That's fewer things that can go wrong.

Faster onboarding

New customer = populate the master tables once, then every future movement flows through the engine automatically.

Cleaner exceptions

When something's wrong, we know instantly whether it's master data (fix config) or transactional (ask the customer).
