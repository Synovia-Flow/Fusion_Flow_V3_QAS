Checkpoint 01 · Normalise · DP-FR-01

Speak one language

The two raw sources use the client's column names and human formats. Normalise rewrites them into the single canonical vocabulary the rest of the platform uses.

  1. Map columns
    Source headers → canonical TSS field names, via the maps in mapping.py. The first non-empty alias wins, so a later blank never wipes a good value.
  2. Fix the clock
    Relative dates like "Tomorrow's Date / 06:30" resolve to strict DD/MM/YYYY HH:MM:SS UTC, plus a parallel sortable timestamp.
  3. Tidy codes
    Trim whitespace; upper-case EORIs, country and port codes.
In

arrival_date_time:
  "Tomorrow's Date / 06:30"
nationality: "gb"

↓ normalise
Out · canonical

arrival_date_time = 26/06/2026 06:30:00
nationality_of_transport = GB

→ NORMALISED
Checkpoint 02 · Enrich · DP-FR-02 / 08

Add what TSS demands

Two jobs: resolve coded values against the cached choice-value reference sets in CFG.Choice_Value_Cache, then apply Birkdale's business rules — the hard-won constants that stop TSS rejecting the payload.

Choice resolution · Rule 9

Before any lookup the runner introspects the cache's real column names via INFORMATION_SCHEMA — names vary between reference sets, so nothing is assumed.

Checkpoint 03 · Construct · DP-FR-03

Build the object graph

The flat, normalised fields are assembled into the shape TSS expects, honouring the cardinality exactly: one header → many consignments → up to 99 goods, each with its own nested arrays.

Rows are written to the 13 PRS tables with an upsert on MovementKey — so re-processing the same movement updates in place rather than duplicating.

Keys

Surrogate …RowID PKs; business keys on MovementKey & ordinals.

Checkpoint 04 · Validate · DP-FR-04

Pass, or fail with a reason

Three layers of checks run against the constructed movement. The first failure isn't fatal — every failing rule is collected so the reject reason is complete.

  1. Always-mandatory
    The fields TSS requires on every header, consignment and goods line.
  2. Conditional
    Rules that fire on a value — e.g. Air needs conveyance_ref; controlled goods need commodity_code.
  3. Movement-type · 3a set
    RoRo Accompanied makes the carrier address block & place fields mandatory (Rule 3).
Pass

All checks clear → header & children marked VALIDATED and persisted, ready for staging.

Fail

Status REJECTED, the reason written to RejectReason and LOG.Error_Log:

header.carrier_name mandatory for movement_type 3a; consignment.container_indicator required …

Trust · DP-FR-06 · Field-level audit

Every change explains itself

Whenever a value is set, changed or cleared, one row goes to EXC.Data_Processing_Enhancement — old value, new value, the rule, the TransactionID. Here is a real trace of movement 2026-06-26|ICR123.

 EXC.Data_Processing_Enhancement  ·  MK=2026-06-26|ICR123
arrival_date_time "Tomorrow's Date / 06:30" → 26/06/2026 06:30:00 [Rule 4]
arrival_port ∅ → GBAUBELBELBEL [Rule 12]
transport_charges ∅ → Y [Rule 11]
goods_domestic_status ∅ → D [Rule 10]
importer_eori ∅ → XI379692092000 [Rule 13]
consignor_eori ∅ → XI379692092000 [Rule 13]
TXN ENS_HEADER NORMALISING → ENRICHING → CONSTRUCTING → VALIDATING → VALIDATED

Only genuine changes are logged — no-op writes are skipped — so the table is a true field-level changelog, not noise.

Operations · Safe to run again

Re-runs don't double up

Mode · latest

Picks up the most recent INGESTED rows that don't yet have a VALIDATED / REJECTED PRS header. The normal scheduled path — process what's new.

Mode · explicit ExecutionID

Re-processes a specific load. CONSTRUCT upserts on MovementKey and ordinals, so rows update in place.

The guarantee

Because the audit only logs real changes, a clean re-run produces zero spurious enhancement rows. And PROCESSING_DRY_RUN=1 runs the whole pipeline, prints everything, and writes nothing — perfect for a pre-flight check.

The build · Separation of concerns & what's next

Where the logic lives

mapping.py · pure data

Column maps, BKD constants, validation rule tables. Change a rule here — never in the engine.

process_data.py · the engine

The four stages, the DB adapter, the audit writer. Imports mapping.py; talks to the DB.

010 + 011 · SQL

The 13 PRS tables, and the run-control parameters seeded into CFG (insert-only, operator-owned).

Quality gate

Delivered by an Architect → PM → 3 Dev → QA team. QA caught four real defects — a kwarg mismatch, a flat-vs-nested rule bug that silenced Rules 10/11/12, a conditional-rule crash and a source-alias clobber — all fixed and re-verified before commit.

One open question

The BKD ENS source doesn't carry the carrier address block or container_indicator, yet a 3a movement requires them. Where should those come from — a fixed carrier config row, or another source field? That's the decision before the first live end-to-end run.
